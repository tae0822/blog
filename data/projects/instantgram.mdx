My primary focus was on **building a scalable system architecture**—separating concerns by decoupling the UI from the data fetching logic using a Service Layer. ##  2. Architecture & Key Features

####  Layered Architecture (API & Service Folders)
To avoid bloated components and messy API routes, I separated the logic into a dedicated **Service Layer**.

- **Service Layer**: Handles direct interactions with Sanity using GROQ queries.
- **API Routes**: Act as a bridge between the client-side components and the services.
- **Impact**: This structure allows the backend to be replaced or modified without impacting UI components.

#### 🔐 Secure Authentication & Access Control
- **NextAuth.js**: Implemented seamless Google Login and session management.
- **Middleware**: Created a custom middleware to protect private routes. Unauthenticated users are automatically redirected to the sign-in page, ensuring that only authenticated users can access protected routes.

#### 👥 Social Interactions & Engagement
- **Following System**: Real-time UI updates for following/unfollowing users.
- **Feed & Discovery**: Implemented a **Multi-Carousel** for followed users and a performant user search feature with optimized query filtering
- **Engagement**: Implemented core social interactions including liking, bookmarking, and commenting with immediate UI feedback.

#### 👤 Profile & Deep-Linking
Dedicated User Profile pages display posts categorized by: **Uploaded**, **Liked**, and **Saved (Bookmarked)**. This required efficient data filtering and optimized GROQ queries to ensure fast loading times.

#### ⚡ Performance Optimization
- Leveraged Next.js App Router for improved routing and rendering performance.
- Optimized image delivery using Next/Image and CDN strategies to reduce load time.

## 🔥 4. Engineering Challenges & Solutions

#### "The Challenge of Data Relationality in a Headless CMS"
Managing complex relationships (e.g., *"Does the current user like this post?"*) in a NoSQL-based CMS like Sanity was challenging. 

**Solution**: I optimized my **GROQ queries** to fetch relational data in a single request and utilized **TypeScript Interfaces** to ensure type safety across the entire data flow, reducing runtime errors significantly.
