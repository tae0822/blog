## 2. Architecture & Key Features

#### 📑 Hybrid Content Management (MD vs. MDX)
I designed a hybrid content system where Markdown is used for simple posts, while MDX is used for technical articles requiring interactive components. This allowed me to maintain fast writing workflows while enabling rich, component-driven documentation when needed.

- **Markdown (.md)**: Used for standard blog posts to ensure maximum writing speed and simplicity.
- **MDX (.mdx)**: Reserved for project deep-dives, allowing the embedding of Custom React Components directly within the documentation.
- **Contentlayer**: Utilized to transform content into type-safe JSON data, making it easy to query and render with full TypeScript support.

#### 🔍 Strategic SEO & Performance
- **Metadata API**: Implemented dynamic SEO for every page. Each post automatically generates unique meta tags.
- **Performance**: Contentlayer generates typed data from markdown files, eliminating runtime errors.

#### ✉️ Full-Stack Communication (Nodemailer)
- **Direct Mail Service**: Built a custom API route using Nodemailer to handle contact form submissions.
- **Serverless Integration**: Developed a secure backend flow to send inquiries directly to personal email.

#### 🌙 Advanced Theme Engineering
- **Next-Themes**: Implemented a flicker-free Dark Mode that respects system preferences.
- **Hydration Stability**: Prevented hydration mismatches by deferring theme rendering until client mount.

## 🔥 4. Engineering Challenges & Solutions

#### Solving the Hydration Mismatch in Themed Layouts
When using next-themes with the App Router, the mismatch between the server-rendered HTML and the client’s theme preference often caused UI glitches or console errors.

**Solution**: I implemented a **Mounted State Pattern** within a custom wrapper and used the `suppressHydrationWarning` attribute on the `<html>` tag. This ensured that the theme provider only synchronized after the client-side mount.
